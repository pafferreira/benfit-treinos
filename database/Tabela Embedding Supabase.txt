Plano: Base de Conhecimento Compartilhada (Shared Embeddings)

Contexto
O Benfit Coach atualmente usa b_benfit_embeddings exclusivamente para memórias privadas de cada usuário. Perguntas factuais como "quais exercícios de bíceps temos?" ou "como fortalecer quadríceps?" são respondidas via Gemini (custo de tokens) ou pelo tryLocalResponse() com texto plano do catálogo.

Objetivo: criar uma base de conhecimento compartilhada (embeddings vetoriais) que atende a todos os usuários, otimiza tokens e abre caminho para templates de treino gerados pela IA.

Decisão Arquitetural: Tabela Separada
Usar b_shared_knowledge (nova tabela), não modificar b_benfit_embeddings.

Razões:

user_id NOT NULL em b_benfit_embeddings exigiria migration breaking para admitir linhas públicas
RLS atual (WHERE user_id = auth.uid()) ficaria ambíguo com escopo público
Índices HNSW separados → scan do catálogo não polui busca de memórias pessoais
Escrita admin-only isolada (service_role) sem risco de conflito com dados de usuário

Passo 1 — SQL: supabase/migrations/add_shared_knowledge.sql
Tabela b_shared_knowledge
CREATE TABLE IF NOT EXISTS public.b_shared_knowledge (
    id             uuid DEFAULT gen_random_uuid() PRIMARY KEY,
    knowledge_type text NOT NULL DEFAULT 'exercise',
    -- Valores: 'exercise' | 'workout_template' | 'fitness_tip' | 'faq'
    source_id      uuid,          -- FK opcional para b_exercises.id, b_workouts.id
    content        text NOT NULL, -- texto descritivo para busca semântica
    metadata       jsonb,         -- { muscle_group, equipment, difficulty, name... }
    embedding      vector(3072) NOT NULL,
    created_at     timestamptz DEFAULT now() NOT NULL,
    updated_at     timestamptz DEFAULT now() NOT NULL
);
